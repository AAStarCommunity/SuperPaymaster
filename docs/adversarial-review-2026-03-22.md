# SuperPaymaster 对抗性审计报告（2026-03-22）

## 视角与目标
本报告以“主网对抗环境”为前提，站在攻击者与失败场景角度复审协议的安全性、经济性、可扩展性与运维合规。未运行测试或自动化扫描，结论基于静态阅读与系统性推演。

## 结论
当前合约体系具备完整功能，但**未达到严格主网发布门槛**。关键问题集中在：
1. 逻辑一致性缺口导致状态视图不可信。
2. 价格路径与外部调用容错不足，导致拒绝服务或经济偏差。
3. 运维与文档约束不足，容易被错误配置与 SLA 失败放大为链上事故。

## 攻击面总览
1. 管理权限面：Owner/DAO/多签可升级与配置关键参数。
2. 价格路径面：Chainlink 价格 + 缓存 + Keeper 更新逻辑。
3. 结算路径面：`validatePaymasterUserOp` 与 `postOp` 的结算与退款。
4. 信誉与治理面：DVT/BLS 聚合器与 ReputationSystem 的信任链。
5. 资产面：GToken 质押、xPNTs 余额、债务记录与 SBT 关系。

## 高风险失败模式（对抗性视角）
1. **状态视图被污染**
`Registry.safeMintForRole` 未更新 `userRoles`，导致索引视图不一致，可能在前端/索引器层形成“角色丢失”假象，进而影响权限与风控。

2. **价格计算溢出引发拒绝服务**
`PaymasterBase._calculateTokenCost` 中多重乘法存在溢出风险，在高 gas cap 或异常 decimals 情况下触发 revert，直接导致 paymaster 停摆。

3. **postOp 外部依赖失败引发赔付风险**
`SuperPaymaster.postOp` 直接调用 `recordDebt`，若 token 异常将导致 `postOp` 失败，可能引发赔付或资金不一致。

## 经济攻击与激励风险
1. **池化 slash 风险**
GTokenStaking 使用共享资金池，slash 会波及所有质押者，易引发挤兑或治理攻击。必须在文档与 UI 明确风险披露。

2. **小额用户费用过高**
`minFee` 与 `feePercent` 叠加对小额用户不公平，可能引发用户放弃退出，进而造成经济声誉风险。

3. **Paymaster 价格路径被操作**
价格缓存 + 容差设置不足可能让攻击者利用 stale price 或极端价格区间获得过低支付成本。

## 可扩展性与链上成本
1. Registry 使用字符串与动态 bytes 作为索引，长期成本高且难以迁移。
2. 多处数组存储成员与角色，规模扩大时查询成本上升。
3. 事件写入过多将增加长期索引成本。

## 运维与合规风险
1. `.env` 与脚本路径不统一，存在配置漂移风险。
2. 缺少自动化门槛（测试、覆盖率、静态扫描）与强制发布流程。
3. 文档多版本并存，缺少单一权威来源与版本映射。

## 关键修复与补强建议
### P0（发布阻断）
1. 修复 `safeMintForRole` 角色数组一致性。
2. 重构 PaymasterV4 价格计算为分段 `mulDiv` 或限制 decimals 与 gas cap。
3. 为 `postOp` 外部调用增加容错与失败告警。

### P1（发布前强烈建议）
1. Oracle 校验补齐，定义 Keeper SLA 与告警策略。
2. 文档明确关键参数安全区间与主网版本映射。
3. 统一配置路径与部署流程，减少操作误差。

### P2（上线后优化）
1. Registry 索引字段逐步迁移为 `bytes32`。
2. 清理零金额 role lock 与冗余数组。
3. 建立 CI 安全门槛与审计记录归档。

## 发布判定建议
当前状态应视为“**准备阶段**”，在完成 P0/P1 之前不建议主网发布。若需提前发布，需至少完成 P0 并建立紧急回滚与多签监控机制。

## 限制说明
本报告为对抗性静态分析，未执行测试或自动化扫描。主网发布前必须补齐测试、覆盖率与静态扫描报告，并完成多签与运维复核。
