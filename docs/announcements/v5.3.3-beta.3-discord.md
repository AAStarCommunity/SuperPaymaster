## 🧵 Follow-up: the H-1 fix explained

```
**H-1: why credit ceiling matters in the debt-fallback path**

In SuperPaymaster, when postOp can't burn xPNTs (user drained their balance mid-UserOp), it falls back to recording a debt. Before beta.3, that fallback path had no credit limit check — an attacker could repeatedly fund → drain → get sponsored indefinitely, accumulating unlimited operator debt.

Fix: the fallback now checks `existingDebt + pendingDebts + thisBill ≤ creditLimit`. Over-ceiling users get auto-blocked and can only be unblocked by the DVT/BLS network (not the contract owner).

This is the "honest poor user" design: users with real xPNTs balance still pay normally. Users deliberately draining get blocked after the first violation.
```

## 🧵 Follow-up: the H-2 fix explained

```
**H-2: the emergency switch that wasn't switching**

xPNTsToken has an `emergencyDisabled` flag that communities can set to halt all token movement. Before beta.3, the `transferFrom` path where `to == msg.sender` (used for self-pull) skipped both the emergency check and the daily rate-limit — meaning a user could bypass the community's emergency stop entirely.

Fix: `to == msg.sender && to != SUPERPAYMASTER_ADDRESS` now routes through the full emergency + rate-limit check. The SP carve-out preserves the depositFor flow.
```
