### 0.3 漏洞链（已严格沿代码走过）

```
T0: operator 持有 ROLE_PAYMASTER_SUPER
    hasRole=true, userRoleCount=1, userRoles=[X], roleMembers[X]∋op

T1: slash 到 0 → Fix-1 fire：仅 hasRole=false
    残留 corrupt: userRoleCount=1, userRoles=[X], roleMembers[X]∋op,
                  roleMetadata, roleSBTTokenIds, (COMMUNITY 时 byName/byENS)

T2: operator 想清理：exitRole(X) → revert RoleNotGranted ← 走不通

T3: operator 重新 registerRole(X)：
    line 235: alreadyHasRole = hasRole[X][op] = false
    line 244: 进入 _firstTimeRegister 分支（语义错误：应进 already-has 分支）
    line 336: roleMembers[X].push(op)  → op 出现 2 次
    line 339: userRoleCount[op]++     → 2 (corrupt)
    line 340: userRoles[op].push(X)   → [X, X] (重复)

T4: operator 正常 exitRole(X)：
    userRoleCount−1 = 1，_removeFromUserRoles 只删一个重复条目
    userRoleCount > 0 → SBT 不被 burn
    operator 永远成"幽灵 SBT 持有者"，
    isEligibleForSponsorship(op) 永远为 true，sponsor 流程持续放行
```

### 0.4 Fix-1 bug 与 H-3 风险对比

| 维度 | H-3（Fix-1 试图修） | Fix-1 引入的新 bug |
|------|-----------|------------|
| 触发前置 | 被 slash 到 0（30% cap × 24h cooldown ≥ 4 轮 / 多日） | 被 slash 到 0 **+** 重新注册 |
| 影响 | 治理瑕疵 — admin 能做的事 multisig 1 tx 可阻止 | **状态永久 corrupt** — userRoleCount / roleMembers / SBT 状态污染 |
| 可逆 | ✅ multisig 即时介入 | ❌ 需 Registry 升级 + 手工 cleanup 工具 |

| 触发自然度 | 被 slash 的人**还想做坏事** | 被 slash 的人**想恢复运营** —— 这是被 slash 后的自然路径，**触发概率更高** |

### 0.7 正确的修复方向（留到 v5.4 + Registry 压缩）

#### 方案 A：在 syncStakeFromStaking 里做完整 cleanup（最干净）

抽出 internal helper `_cleanupRole(roleId, user, releaseStake)`，让 `exitRole` 和 `syncStakeFromStaking` 都调用：

```solidity
function _cleanupRole(bytes32 roleId, address user, bool callerIsStaking) internal {
    hasRole[roleId][user] = false;
    _removeFromRoleMembers(roleId, user);
    if (userRoleCount[user] > 0) userRoleCount[user]--;
    _removeFromUserRoles(user, roleId);

    if (roleId != ROLE_PAYMASTER_AOA && roleId != ROLE_PAYMASTER_SUPER) {
        if (roleId == ROLE_COMMUNITY) { /* clear byName/byENS */ }
        delete roleMetadata[roleId][user];
        delete roleSBTTokenIds[roleId][user];
    }

    if (userRoleCount[user] == 0) {
        if (SUPER_PAYMASTER != address(0)) {
            ISuperPaymaster(SUPER_PAYMASTER).updateSBTStatus(user, false);
        }
        (bool ok,) = address(MYSBT).call(abi.encodeCall(IMySBT.burnSBT, (user)));
        if (!ok) emit SBTBurnFailed(user, roleId);
    }
    // callerIsStaking 路径下不调 unlockAndTransfer（stake 已经 0）
}
```

**字节估算**：抽出共享 helper 后净增 ≈ **80-120B**（具体取决于编译优化）。

**前提**：Registry 当前 24,499B / 余量 77B → 装不下。**必须先做 Registry 压缩 ≥ 150B**：
- 把所有 `revert "string"` 改为 custom error
- 合并重复 internal 函数
- 评估有没有可移出的非热路径函数（如 ENS 兼容相关）

#### 方案 B：保留 hasRole=true，加 `isActiveRole` flag

```solidity
mapping(bytes32 => mapping(address => bool)) public isActiveRole;
// SP._requireSuperOperatorRole 检查 hasRole && isActiveRole
```

**优**：不动数组状态，字节更省
**劣**：双 flag 语义不直观，需要全代码库 audit 哪些地方读 hasRole

#### 方案 C：彻底接受 H-3 为 mitigation，不做合约改动

这是 audit 团队的判定。当前 30% cap + 24h cooldown 给了足够人工干预时间。**这是最稳妥的方案。**

## 一、安全问题（已确认存在缺口，需 v5.4 修复）

### 1.1 M-2：exitRole 不同步 MySBT membership 状态

**状态：未修复（代码确认 + EIP-170 余量不足）**

#### 问题（已确认）

`Registry.exitRole()` 退出角色时，**不调用** `MySBT.deactivateMembership()`。  
仅当用户退出最后一个角色（`userRoleCount == 0`）时才调用 `burnSBT`。  
当用户退出某一角色但保留其他角色时，MySBT 中对应社区的 membership 仍处于 `isActive = true` 状态。

#### 影响

- 前端/链下查询 MySBT 状态会看到错误结果（已退出的 community 仍显示 active）
- `recordActivity()` 对退出了该 community 的 ENDUSER 仍可写入活动记录（已有 M-2A 部分缓解，见 #168）
- **不影响资金安全**

#### 为何仍未修复 —— EIP-170 余量分析（v5.4 关键问题）

Registry 当前字节数（已实测）：

```
Registry runtime: 24,353 bytes
EIP-170 上限:     24,576 bytes
当前余量:            223 bytes  ← 不够
M-2 修复需要:      ~320 bytes  ← 超出余量约 97 bytes
```

**223B 余量 < 320B 需求，无法直接加入。**  
必须先对 Registry 再压缩 ≥ 100 bytes，才能安全落入 M-2 修复。

> v5.3.1 发布时 Registry 已经经历了多轮压缩（PR #167 记录）。  
> 当前 223B 余量属于"安全垫"，不足以容纳 M-2 改动。

#### v5.4 修复步骤

① **先压缩 Registry**（目标：释放 ≥ 150B 余量）
- 候选项：将较长的 `revert` 字符串改为自定义 error；合并重复的 internal 逻辑
- 每次改动后用 `forge build && python3 -c "import json,os; d=json.load(open('out/Registry.sol/Registry.json')); print(len(bytes.fromhex(d['deployedBytecode']['object'][2:])))"`  验证

② **加入 deactivateMembership 调用**（目标代码，加在 `exitRole` 末尾 stake 返还之后）：

```solidity
// 退出 ENDUSER 角色时，同步注销 MySBT membership（非阻塞）
if (roleId == ROLE_ENDUSER && address(MYSBT) != address(0)) {
    bytes memory meta = roleMetadata[roleId][msg.sender];
    if (meta.length >= 32) {
        (address community,) = abi.decode(meta, (address, bytes));
        (bool ok,) = address(MYSBT).call(
            abi.encodeCall(IMySBT.deactivateMembership, (msg.sender, community))
        );
        if (!ok) emit MembershipDeactivateFailed(msg.sender, roleId);
    }
}
```

③ 确认 `IMySBT` 接口已声明 `deactivateMembership(address user, address community) external`

④ 补测试：`exitRole` 后验证 `MySBT.membershipStatus(user, community).isActive == false`

### 1.2 H-3：slash 后 Registry.hasRole 布尔值未撤销

**状态：未完全修复，有接受的缓解措施**

#### "接受缓解"的确切含义

这个问题**存在**，安全团队评估后认为当前风险可接受（不是"解决了"，而是"风险低到可以暂时不修"）。

#### 问题（代码确认）

`GTokenStaking.slashByDVT()` 将 `lock.amount` 降至 0 时：
- ✅ 清除 `userActiveRoles` 数组（`_removeUserRole`）
- ✅ 通过 `_syncRegistry` 更新 `Registry.roleStakes` 缓存（roleStakes 归零）
- ❌ **不**将 `Registry.hasRole[roleId][operator]` 设为 `false`

#### 实际风险

- `_requireSuperOperatorRole()` 用 `REGISTRY.hasRole(...)` 做权限检查
- 被 slash 到 0 的 operator：roleStakes = 0，但 `hasRole` 仍为 `true`
- 因此 `withdrawDeposit` 等 admin 函数权限 **仍然有效**
- `validatePaymasterUserOp` 用 `config.isConfigured`（非 `hasRole`），gas 赞助权限也未被阻断

#### 已有缓解

- `slashOperator` 每次最多 slash 30%，且有 24h 冷却期
- 将 operator stake 降至 0 至少需要多轮、多天，给足人工干预时间
- SP 端的 `configureOperator` 可随时暂停（`isPaused = true`），一行 tx 即可停止赞助

#### v5.4 修复步骤

① 在 `slashByDVT` 的 `lock.amount == 0` 分支增加：

```solidity
REGISTRY.setHasRole(roleId, operator, false);
emit RoleRevokedBySlash(operator, roleId);
```

② 或者修改 `_requireSuperOperatorRole` 为双重检查：

```solidity
require(
    REGISTRY.hasRole(roleId, msg.sender) &&
    REGISTRY.getEffectiveStake(msg.sender, roleId) >= minStake,
    "insufficient stake"
);
```

③ 补 slash-to-zero 后权限验证测试

## 五、E2E 测试基础设施（工程债）

### 5.1 BLS / DVT "mock" 问题说明

> **用户疑问**："BLS mock 是什么问题？我们不是有实际的合约吗？为什么说是 mock？"

#### 澄清

**真实情况**：DVTValidator 和 BLSAggregator 合约**确实已部署**到 Sepolia，是真实的链上合约，不是 mock。

**为何测试需要 mock**：

`H2-reputation-sync` 测试的调用链：
```
ReputationSystem.syncToRegistry()
  → REGISTRY.batchUpdateGlobalReputation(proposalId, users, scores, epoch, proof)
      → BLSValidator.verifyAggregateSignature(proof, ...)
          → 需要 BLS-12-381 椭圆曲线聚合签名
              ← 签名必须来自 ≥3 个真实运行的 DVT validator 节点
```

问题不在合约，而在**基础设施**：
- BLS-12-381 聚合签名必须由 ≥3 个 DVT 节点运行产生
- 当前没有持续运行的 DVT 节点集群
- 没有集群 → 无法生成合法的 proof → `verifyAggregateSignature` 必然 revert
- 因此 `syncToRegistry` 在 Sepolia 上**无法自动化测试**

**"mock" 在这里的含义**：仅指 Foundry 测试中的 `vm.mockCall`，用于在 Anvil（本地）模拟 BLS 签名验证通过，使 CI 可以跑完全路径，而不需要真实 DVT 集群。这是**测试侧 mock**，不是合约 mock。

#### v5.4 动作

在 Anvil 测试中用 `vm.mockCall` 模拟 `BLSValidator.verifyAggregateSignature` 返回 true，实现 `batchUpdateGlobalReputation` 的全路径 CI 覆盖（不影响 Sepolia 真实合约，只是测试工具）。
