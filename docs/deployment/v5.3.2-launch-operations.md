## 一、本次合约改动（最终版）

经过对 v5.4 TODO 全部条目逐项严格 review，最终落入合约的改动只有两项。其余条目要么已通过其他机制缓解，要么属于运营 / 前端流程问题。

### 1.1 Fix-1：slash → 0 自动吊销 hasRole

**文件**: `contracts/src/core/Registry.sol`，`syncStakeFromStaking`  
**Registry 版本**: 5.3.3 → 5.3.4

**问题**: 原 `slashByDVT` / `slash` 将 stake 降至 0 时，只同步了 `roleStakes` 缓存，未撤销 `hasRole` 布尔值。被 slash 至 0 的 operator 在治理层面已失去资格，但 `_requireSuperOperatorRole` 仍放行其管理动作（`configureOperator` / `setOperatorLimits` / `deposit` / `_validateX402AndComputeFee`）。这不是直接资金风险，但属于治理瑕疵。

**修复**:
```solidity
function syncStakeFromStaking(address user, bytes32 roleId, uint256 newAmount) external {
    if (msg.sender != address(GTOKEN_STAKING)) revert Unauthorized();
    roleStakes[roleId][user] = newAmount;
    // Invariant: hasRole ⇒ stake > 0. When stake drops to 0 revoke the role.
    if (newAmount == 0 && hasRole[roleId][user]) {
        hasRole[roleId][user] = false;
        emit RoleRevoked(roleId, user, msg.sender);
    }
}
```

**为何在 Registry 改而非 GTokenStaking**:
- `syncStakeFromStaking` 已是所有 stake 变更的统一汇聚点（slash / unlock / topUp）
- 已有 `onlyGTokenStaking` 认证，无需新增信任边界
- exitRole 路径下 hasRole 已先被清零，此处幂等
- 复用既有 `RoleRevoked` 事件，避免新增 event 占字节

**语义变化（破坏性，已用测试验证）**:
- **新语义**: slash 到 0 = 强制退出。被 slash 用户**不再需要也不能**手动调 `Registry.exitRole`（会被 `RoleNotGranted` 拦截）
- **影响**: 完全合理。被 slash 至 0 的用户 stake = 0，无 GToken 可退；hasRole 自动撤销后他们已被系统排除
- **副作用**: `userRoles[user]` 和 `roleMembers[roleId]` 数组保留陈旧条目（view 函数会看到，但所有权限检查走 hasRole）。v5.5 Registry 压缩时可一并清理

**字节影响**: Registry 24,352B → 24,499B（+147B），EIP-170 余量 **77B**

### 1.2 Fix-4：setXPNTsFactory 零地址防护

**文件**: `contracts/src/paymasters/superpaymaster/v3/SuperPaymaster.sol:458`  
**SuperPaymaster 版本**: 5.3.1 → 5.3.2

**问题**: `setXPNTsFactory` 是 onlyOwner（多签）但缺零地址校验。多签误操作可使 SP 所有 xPNTs 路径静默失败。

**修复**:
```solidity
function setXPNTsFactory(address _factory) external onlyOwner {
    if (_factory == address(0)) revert InvalidAddress();   // ← 新增
    emit XPNTsFactoryUpdated(xpntsFactory, _factory);
    xpntsFactory = _factory;
}
```

**字节影响**: SuperPaymaster 24,239B，余量 **337B**

## 二、未修改项及其依据

本节记录所有审计 TODO 中"未修改"的条目和判定理由，供后续审计追溯。

### 2.1 H-3 / hasRole 双重校验 — 不需要

**判定**: Fix-1 已从源头解决（slash → 0 自动吊销 hasRole）。无需在每个敏感函数加 `hasRole && getEffectiveStake >= minStake` 双重检查。

### 2.2 P1-35 / recordDebt 累计 cap — 不需要

**判定**: 已有完整的异步保护链路：
```
DVT 集群链下监控 user.debt vs Registry.getCreditLimit
   ↓ 检测到超限
BLS 聚合签名（≥3 validator）
   ↓
Registry.updateOperatorBlacklist(BLS proof)
   ↓ 验证后
SuperPaymaster.updateBlockedStatus(userOpState[op][u].isBlocked = true)
   ↓
下一次 validatePaymasterUserOp 第 1059 行 reject
```

设计权衡：链上 validate 不做同步 credit 检查（成本与延迟权衡），由 DVT 周期性扫描兜底。**前提是 DVT 集群在线**，见运营部分。

### 2.3 L-4 / topUpStake CEI — 不需要

**判定**: GToken 是标准 `ERC20Capped + ERC20Burnable`，无 ERC-777 回调，无 `_beforeTokenTransfer` 钩子。`topUpStake` 仅 `onlyRegistry` 可调。无实际 reentrancy 风险。

### 2.4 P1-11 / setAgentRegistries Timelock — 不需要

**判定**: Agent 走 EIP-8004 链上注册，且 postOp 中 `_recordXPNTsDebt` 依然要烧 xPNTs 或记 debt。**不存在"免费赞助"路径**——agent 和 SBT holder 用同一付费路径。即便恶意 registry 让任意人成为"agent"，其 debt 仍受 DVT 异步监控约束。Operator 自担有限风险。

### 2.5 M-2 / exitRole 不同步 MySBT — 已自然缓解

**判定**:
- M-2A 已在 `MySBT.recordActivity` 加 `hasRole(ROLE_ENDUSER, u)` 交叉检查，**阻断了最危险路径**（声誉刷量）
- MySBT 暴露 `leaveCommunity(address)` 用户自助调用，可自行将 `isActive = false`
- 剩余只有"前端查询时 isActive 字段瞬时陈旧"的纯 UX 问题
- SDK 应在 `Registry.exitRole` 之后引导用户调用 `MySBT.leaveCommunity`

### 2.6 P1-21 / operator → paymaster 永久绑定 — 推迟

**判定**: 架构层议题，无资金风险。如未来出现 operator 强制注销场景再追加 `revokePaymaster`。

## 四、运营保障（关键）

### 4.1 DVT 集群必须 24/7 运行 ⭐

**为什么**: 本系统的 credit 超额保护是**异步**的——链上 validate 不做同步检查，依靠 DVT 集群周期性扫描后通过 `updateOperatorBlacklist` 拦截。

**最低要求**:
- ≥ 3 个独立 BLS validator 节点持续在线
- 每个用户 debt 监控周期 ≤ 5 分钟
- 任何节点离线告警 → 1 小时内人工恢复

**最坏情况**: DVT 集群完全停摆，恶意用户可在停摆窗口期内积累 debt（仍受单笔 `maxSingleTxLimit` 约束）。Operator 应监控自己的 `aPNTsBalance` 异常下降。

**监控指标**:
- DVT 节点存活数（< 3 触发告警）
- `Registry.batchUpdateGlobalReputation` 链上调用频率（每用户每天应 ≥ 1 次）
- 各 operator `aPNTsBalance` 下降速率（异常陡降触发告警）

### 4.2 价格 Keeper 必须在线 ⭐

`SuperPaymaster.cachedPrice` 由 SDK keeper 周期更新，过期会导致 `validatePaymasterUserOp` 返回 `validUntil < block.timestamp` → bundler 拒收。

- Keeper 调用周期 ≤ `priceStalenessThreshold / 2`（默认 300s，建议每 120s 调一次）
- Chainlink staleness 兜底已通过 P0-10 `EMERGENCY_TIMELOCK = 1 hours` 实现，紧急覆盖需 owner 介入

### 4.3 多签操作纪律

**onlyOwner 函数清单**（所有调用必须走多签）:
- `Registry.setStaking / setMySBT / setSuperPaymaster / setBLSAggregator`
- `Registry.setCreditTier / setLevelThresholds / setReputationSource`
- `SuperPaymaster.setXPNTsFactory / setTreasury / setAPNTsToken (有 7 天 timelock)`
- `SuperPaymaster.setPriceStalenessThreshold / setMaxGasCostCap`
- `SuperPaymaster.setAgentRegistries`（Agent 主网上线前 review；当前低风险）
- `SuperPaymaster.queueBLSAggregator / executeBLSAggregatorUpdate`（24h timelock）
- `BLSValidator.setMinThreshold / setDefaultThreshold`
- `GTokenStaking.setExitFee / setMaxStakePerRole`
- `xPNTsFactory` 的 owner-only 函数
- 所有 paymaster 实例的 `withdrawProtocolRevenue`

**操作纪律**:
- 任何参数变更必须有 GitHub Issue 或 governance proposal 记录
- 多签 propose 后等待 ≥ 24h（即使合约无 timelock 也走人工冷静期）
- 单一签名人禁止持有多个 multisig 助记词

### 4.4 Operator 自查项

社区 operator（运营 SuperPaymaster 实例的 community）每周应 check：
- 自己的 `operators[me].aPNTsBalance` 下降速率
- 自己 community 的 `getCommunityByName` / `getCreditLimit` 设置仍符合预期
- 自己的 `config.isPaused` 状态
- `paymasterByOperator[me]`（AOA 模式）仍指向已部署的 paymaster
- ⚠️ 若怀疑自己 stake 被 slash，调 `registry.hasRole(ROLE_PAYMASTER_SUPER, me)` 自查——本次 Fix-1 后，stake 归零即 hasRole 自动撤销

异常时立即调用 `configureOperator(... isPaused = true)` 暂停赞助，等待人工介入。

### 4.5 紧急响应剧本

| 场景 | 第一响应 | 后续 |
|------|---------|------|
| Operator 私钥泄露 | Operator: `configureOperator(...isPaused=true)` | DAO 通过 `slashOperator` 进一步约束 |
| DVT 节点全部离线 | Ops 团队：暂停受影响 operator | 恢复 ≥3 节点上线，重新启动监控 |
| Chainlink feed 停更 | 触发 `EMERGENCY_TIMELOCK` 紧急价格 | 1 小时窗口内必须人工确认价格 |
| xPNTs 大规模异常铸造 | xPNTs `emergencyDisabled = true` | 调查后通过 governance 恢复 |
| Registry / SP 升级 bug | 暂停所有 SP 实例（每个 operator 单独 isPaused） | 通过 UUPS 回滚 implementation |

### 4.6 主网上线节奏建议

**Stage 1（前 7 天）**: Soft launch
- 仅开放 1-3 个 trusted community operator 注册
- 每日活跃用户上限 100
- DVT 全人工 review，先不开 auto-block

**Stage 2（第 8-30 天）**: 受控放量
- 开放公开 operator 注册（multisig 审核）
- 每日活跃用户上限 1,000
- DVT auto-block 启用，但保留人工 override

**Stage 3（30 天+）**: 完全开放
- 移除每日活跃上限
- DVT 全自动运行
- 每月 governance review credit tier 与 levelThresholds 参数
