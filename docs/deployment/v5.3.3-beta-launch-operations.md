## 二、deploy-core 完整流程（v5.3.3-beta 起）

```
./deploy-core <env> [--force] [--fresh-deploy]
        │
        ├─ Phase 1: 部署或升级合约
        │   ├─ anvil         → DeployAnvil.s.sol    (每次全新)
        │   ├─ 有 registry proxy → UpgradeLive.s.sol   (UUPS 保留状态)
        │   └─ 无 proxy / --fresh-deploy → DeployLive.s.sol  (⚠️ 需确认)
        │
        ├─ Phase 2: 调用 ./audit-core <env> --force
        │   ├─ A) forge script 功能检查 (7 项 Check)
        │   ├─ B) ABI 选择器对比 (本地 ABI vs 链上 bytecode)
        │   └─ C) 接口抽查 (cast call 关键 view 函数)
        │
        └─ Phase 3: Etherscan 验证 (仅 live 网络，部署后)
```

### UUPS 升级策略（从 v5.3.3-beta 起强制）

**任何对 SuperPaymaster 或 Registry 的修改，必须通过 UpgradeLive.s.sol，不得 re-deploy proxy。**

```bash
# 正常升级（code hash 变了会自动触发）
./deploy-core sepolia

# 强制升级（hash 未变但需要重跑，如手动 patch 后）
./deploy-core sepolia --force

# ⚠️ 仅限全新网络首次部署（会丢失所有链上状态）
./deploy-core <new-network> --fresh-deploy
```

## 三、audit-core 三维度说明

### A) Forge Script 功能检查

运行 7 个 Check 脚本，验证合约状态、版本号、wiring、社区注册：

```
Check04_Registry   — Registry 版本、staking、MySBT、credit tier
Check01_GToken     — GToken 版本、供应量、owner
Check02_GTokenStaking — staking 版本、total staked
Check03_MySBT      — SBT 版本、oracle
Check07_SuperPaymaster — SP 版本、entryPoint、priceFeed、operator config
Check08_Wiring     — SP ↔ Registry wiring、agent registries
VerifyV3_1_1       — 跨合约整体验证（部署者 operator balance 等）
```

### B) ABI 选择器对比

将本地 `abis/SuperPaymaster.json` 和 `abis/Registry.json` 中的每个 public function，计算其 4-byte selector，逐一检查是否存在于链上 bytecode 中。

**失败含义**: 链上部署的是旧版本 bytecode，或本地 ABI 与实际部署不匹配（ABI drift）。

### C) 接口抽查（cast call）

| 调用 | 期望 | 含义 |
|------|------|------|
| `SP.version()` | 含 "SuperPaymaster" | proxy 指向正确 impl |
| `SP.entryPoint()` | 含 "0x00000000717" | EntryPoint 地址正确 |
| `SP.paused()` | 不 revert | 合约可正常响应 |
| `Registry.version()` | 含 "Registry" | Registry impl 正确 |
| `Registry.owner()` | 不 revert | 所有权完整 |
| `MicroPaymentChannel.owner()` | 不 revert（若已部署） | MC 合约存活 |

### 幂等机制

每个 check 单独有 stamp 文件（`deployments/.audit.<env>.<check>.stamp`），内容是 `registryImpl|spImpl|updateTime` 指纹：
- **通过后立刻写 stamp** → 下次跳过该 check
- **部署后 `deploy-core` 传 `--force`** → 所有 stamp 强制失效，全部重跑
- **手动重跑单项**: `./audit-core sepolia --force`

### RPC 重试

每个 check 最多重试 3 次（退避 5s / 10s），check 之间 sleep 2s，应对 Alchemy 300 RPS burst 限制导致的 "Connection reset by peer"。

## 四、Sepolia vs 主网的差异

### 4.1 签名方式

| 环境 | 签名方式 | 说明 |
|------|---------|------|
| **Sepolia** | `PRIVATE_KEY` 明文（`.env.sepolia`） | 测试网可接受，方便 CI/自动化 |
| **主网 (OP/Mainnet)** | `DEPLOYER_ACCOUNT` Foundry keystore | `cast wallet import` 导入加密 keystore；明文私钥**绝对禁止** |

主网 `.env.optimism` / `.env.mainnet` 应只有：
```bash
DEPLOYER_ACCOUNT=mainnet-deployer   # Foundry keystore 名称
DEPLOYER_ADDRESS=0x<multisig或部署者EOA>
RPC_URL=https://...
OPTIMISM_RPC_URL=https://...
# 不得出现 PRIVATE_KEY / PRIVATE_KEY_JASON 等明文私钥
```

### 4.2 部署后 prepare-test

| 操作 | Sepolia | 主网 |
|------|---------|------|
| `./prepare-test <env>` | ✅ 运行（注册 Anni 角色、部署 V4 paymaster、刷新价格缓存） | ❌ **禁止**（测试账户注册不得出现在主网） |
| `forge script RegisterEnduser.s.sol` | ✅ E2E 测试前运行 | ❌ **禁止**（ENDUSER 用户自助注册） |
| `node setup-gasless.js` | ✅ E2E 前置 | ❌ **禁止** |

### 4.3 MicroPaymentChannel constructor

`new MicroPaymentChannel(deployer)` — constructor 参数是 `owner`。

- **Sepolia**: `deployer = Jason EOA (0xb5600060...)` — 可接受
- **主网**: `deployer` 应为**多签合约地址**，不应是个人 EOA。部署前确认：
  ```bash
  # 确认 DEPLOYER_ADDRESS 指向多签
  cast call $MULTISIG "getOwners()(address[])" --rpc-url $RPC_URL
  ```
  如果主网用个人 EOA 部署，部署后**立即** `transferOwnership(multisig)`。

### 4.4 audit-core 在主网的注意事项

- **Alchemy 主网 key** 和 Sepolia 不同，确保 `.env.optimism` 里 `OPTIMISM_RPC_URL` 已设置
- ABI selector 检查依赖 `abis/` 目录下的 JSON，需在部署前运行 `./sync_to_sdk.sh` 或 `forge inspect` 更新
- 主网 RPC 通常更稳定，但 `MAX_RETRIES=3` 仍保留作为保险

### 4.5 Etherscan / Blockscout 验证

| 环境 | 验证目标 | API Key 位置 |
|------|---------|-------------|
| Sepolia | Etherscan Sepolia | `ETHERSCAN_API_KEY` in `.env.sepolia` |
| OP Mainnet | Blockscout / Etherscan OP | `ETHERSCAN_API_KEY` in `.env.optimism`（需 OP Etherscan key）|
| Mainnet | Etherscan | `ETHERSCAN_API_KEY` in `.env.mainnet` |

Phase 3 (`verify-all.sh`) 在 `deploy-core` 里是 non-fatal（失败打 warning，不终止）。主网部署后**必须手动确认所有合约在 Etherscan 上已验证**，否则社区审计和 SDK 用户无法读 ABI。

## 五、主网部署完整 Checklist

### 5.1 部署前

- [ ] `forge test` 全量通过（当前 923 项）
- [ ] Sepolia E2E 全绿（22 项）
- [ ] `forge inspect SuperPaymaster bytecode` 字节数 < 24,576B
- [ ] `abis/` 目录已同步最新 ABI（`./sync_to_sdk.sh` 或手动 `forge inspect`）
- [ ] `.env.optimism` 中 `DEPLOYER_ACCOUNT` 指向 Foundry keystore（**无明文私钥**）
- [ ] 多签签名人就位（至少 M-of-N 已在线）
- [ ] `config.optimism.json` 已存在 registry proxy 地址（UUPS 升级路径）
- [ ] `PRIVATE_KEY_ANNI` 等测试密钥**不在** `.env.optimism` 中

### 5.2 部署

```bash
source .env.optimism
./deploy-core optimism
# 如需强制重新部署 impl（hash 未变）:
./deploy-core optimism --force
```

deploy-core 会自动：
1. 运行 UpgradeLive.s.sol（UUPS 升级 SP + Registry，幂等部署 MC）
2. 运行 audit-core（三维度验证）
3. 运行 verify-all.sh（Etherscan 验证）

### 5.3 部署后手动确认

```bash
# 版本号
cast call $SP_PROXY   "version()(string)" --rpc-url $OPTIMISM_RPC_URL
cast call $REG_PROXY  "version()(string)" --rpc-url $OPTIMISM_RPC_URL

# MicroPaymentChannel owner（若首次部署，应为部署者；主网需转给多签）
cast call $MC_ADDR "owner()(address)" --rpc-url $OPTIMISM_RPC_URL

# transferOwnership to multisig（若需要）
cast send $MC_ADDR "transferOwnership(address)" $MULTISIG \
  --rpc-url $OPTIMISM_RPC_URL --account $DEPLOYER_ACCOUNT
```

### 5.4 不需要做的事（主网禁止）

- ❌ `./prepare-test optimism`
- ❌ `forge script RegisterEnduser.s.sol` 对主网
- ❌ `./deploy-core optimism --fresh-deploy`（除非是全新网络首次）
- ❌ 修改 `PRIVATE_KEY_ANNI` 或任何测试账户私钥相关配置
