## 1. Objective

Make SuperPaymaster natively discoverable by AI Agents and natively integrated with the ERC-8004 Agent identity standard. Three workstreams:

1. **On-chain**: ERC-8004 identity verification + reputation-driven sponsorship policies
2. **Off-chain**: SKILL.md hosting + OpenAPI x-payment-info + MCP payment signal
3. **Tooling**: `@superpaymaster/cli` command-line tool for Agent/developer use

**Dependency**: V5.3 on-chain changes (new view functions + config) can develop in **parallel** with V5.1/V5.2 since they only add new functions. Integration requires V5.1+V5.2 merge first.

#### T2: `isRegisteredAgent()` — Agent Identity Check (Week 1)

```solidity
function isRegisteredAgent(address account) public view returns (bool) {
    if (agentIdentityRegistry == address(0)) return false;
    try IAgentIdentityRegistry(agentIdentityRegistry).balanceOf(account) returns (uint256 bal) {
        return bal > 0;
    } catch {
        return false;
    }
}
```

#### T3: `isEligibleForSponsorship()` — Dual-Channel Identity (Week 2)

**Description**: Combine existing SBT check with ERC-8004 check for gas sponsorship eligibility.

```solidity
function isEligibleForSponsorship(address user, address operator) public view returns (bool) {
    // Channel 1: Traditional — user has SBT via our Registry
    bool hasSBT = sbtHolders[user];
    // Channel 2: ERC-8004 — user is a registered Agent
    bool isAgent = isRegisteredAgent(user);
    return hasSBT || isAgent;
}
```

#### T4: Agent Sponsorship Policies (Week 2-3)

**Description**: Reputation-driven, per-operator sponsorship tiers for ERC-8004 agents.

struct AgentSponsorshipPolicy {
    uint256 minReputationScore;  // Minimum ERC-8004 reputation
    uint256 sponsorshipBPS;      // Rate (10000 = 100%)

| T4.6 | Apply sponsorship rate in `_consumeCredit()` for ERC-8004 agents | 2h | SuperPaymaster.sol |
| T4.7 | Daily cap tracking: `agentDailySpend[address][uint256]` mapping (消耗 1 storage slot from __gap) | 1h | SuperPaymaster.sol |
| T4.8 | Unit tests: tier matching, daily cap enforcement, default policy | 4h | test/ |

**Policy Matching Logic**:

```
1. Get agent's ERC-8004 reputation score
2. Find the highest-tier policy where score >= minReputationScore
3. Apply sponsorshipBPS to the charge amount
4. Check dailySpend + newCharge <= maxDailyUSD
```

**Example Configuration**:

```
Operator sets 3 tiers:
  Tier 0: score >= 80 → 100% sponsorship, $10/day cap
  Tier 1: score >= 50 → 50% sponsorship, $5/day cap
  Tier 2: score >= 0  → 10% sponsorship, $1/day cap (new agents)
```

**Acceptance Criteria**:

- [ ] Operator can set multi-tier policies
- [ ] Agent matched to correct tier based on reputation
- [ ] Sponsorship rate correctly applied to charge amount
- [ ] Daily cap enforced per agent per UTC day
- [ ] No policies set → full charge (no sponsorship for ERC-8004 agents)
- [ ] Agent with SBT → existing sponsorship (policies not applied)
- [ ] Storage layout verified

#### T5: Sponsorship Feedback to ERC-8004 (Week 3)

**Description**: After successful gas sponsorship, submit positive feedback to ERC-8004 Reputation Registry.

```solidity
function _submitSponsorshipFeedback(uint256 agentId, uint256 usdAmount) internal {
    if (agentReputationRegistry == address(0)) return;
    try IAgentReputationRegistry(agentReputationRegistry).giveFeedback(
        agentId,
        int128(int256(usdAmount)),
        18,                 // decimals
        "gas-sponsor",      // tag1
        "success",          // tag2
        "",                 // endpoint
        "",                 // feedbackURI
        bytes32(0)          // No file hash
    ) {} catch {}
}
```

**Sub-tasks**:

| # | Task | Est. | File |
|---|------|------|------|
| T5.1 | Implement `_submitSponsorshipFeedback()` internal function | 1h | SuperPaymaster.sol |
| T5.2 | Call from `_consumeCredit()` when user is ERC-8004 agent | 1h | SuperPaymaster.sol |
| T5.3 | Determine agentId from address (reverse lookup or require parameter) | 1h | SuperPaymaster.sol |
| T5.4 | Make feedback submission optional (gas-conscious: skip if gas > threshold) | 1h | SuperPaymaster.sol |
| T5.5 | Unit tests with mock Reputation Registry | 2h | test/ |

**Design Decision**: Feedback is wrapped in try-catch so it never causes the main transaction to revert. If gas cost is a concern, make it operator-configurable (`enableFeedback` flag).

**Acceptance Criteria**:

- [ ] Successful sponsorship → feedback submitted to ERC-8004
- [ ] Reputation registry not configured → silently skip
- [ ] Reputation registry reverts → silently skip (no impact on sponsorship)
- [ ] Feedback contains correct agentId, value, and tags
