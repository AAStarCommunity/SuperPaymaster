## 评级标准

| 严重度 | 业务影响 | 利用难度 | 处理时限 |
|---|---|---|---|
| **Critical** | 资金可被盗 / 治理被夺 / 服务永久不可用 | 任意攻击者可触发 | 上主网前必修 |
| **High** | 资金/状态可被特定角色操纵 / 关键 Liveness 失败 | 需特权或特定时序 | 上主网前必修 |
| **Medium** | 经济激励偏离设计 / 误用导致资金损失 | 需用户配合或经济成本 | 上主网前应修 |
| **Low** | 代码质量 / 不变量缺保护 / 边界处理不全 | 不直接危害 | 上主网后 1–2 迭代修 |
| **Info** | 文档/事件缺失 / 命名 / 可读性 | 无 | 顺手清理 |

**风险系数** = `业务影响权重 × 利用可行性`（业务: 5/4/3/2/1, 利用: 4/3/2/1）。Critical = 16–20, High = 9–15, Medium = 5–8, Low = 2–4, Info = 1。

### 1.2 角色状态机（简表，详细图见附录 A）

```
COMMUNITY (stakeAmount=0 → ticket-only; >0 → ticket+stake)
  registerRole/safeMintForRole → ACTIVE
  ACTIVE → exitRole → EXITED (退还 net stake; ticket fee 已支付)
  ACTIVE → 可 batchUpdateGlobalReputation 反馈

ENDUSER (stakeAmount=0 默认；>0 高级用户)
  registerRole(self) → ACTIVE (sbtHolders=true, 可 sponsored)
  ACTIVE → 可重复 registerRole（更新 metadata + stake topUp）
  ACTIVE → exitRole → EXITED (sbtHolders=false 当 userRoleCount→0)
  ACTIVE → updateSBTStatus(false) → 不可 sponsored（保留 SBT）

PAYMASTER_SUPER / PAYMASTER_AOA
  必须先 hasRole[COMMUNITY]
  registerRole → ACTIVE
  ACTIVE → SP.deposit/withdraw aPNTs；validatePaymasterUserOp 主路径
  ACTIVE → BLS slashOperator → MINOR(10%)/MAJOR(全 + paused)
  ACTIVE → exitRole → EXITED（注意：SP.operators 内的 aPNTsBalance 不会被清零，需要先 SP.withdraw）

DVT / ANODE / KMS
  registerRole + 锁仓 → ACTIVE
  ACTIVE → DVTValidator.slashByDVT → 比例减少
  ACTIVE → exitRole → EXITED（lockDuration 校验）
```

### 1.3 价值流图（核心路径）

**A. 用户首次 sponsored userOp（SP, AOA+ 模式）**
```
User AA (xPNTs balance) ─┐
                          │
EntryPoint validate ──────┼─── SP.validatePaymasterUserOp
                          │      ├─ 校验 sbtHolders / Agent NFT
                          │      ├─ operator.aPNTsBalance -= initialAPNTs(1.2x)
                          │      └─ protocolRevenue += initialAPNTs
                          │
EntryPoint execute (gas)  │      (SP EntryPoint deposit ETH consumed by op)
                          │
EntryPoint postOp ────────┼─── SP.postOp
                          │      ├─ finalCharge = actualBase * 1.1
                          │      ├─ refund = initialAPNTs - finalCharge → operator
                          │      ├─ protocolRevenue -= refund (净: +finalCharge)
                          │      └─ user xPNTs burn (finalCharge * exchangeRate)
                          │           └─ 失败 → recordDebt → pendingDebts
                          ▼
                  Owner withdrawProtocolRevenue → treasury
```

**B. x402 Direct Settlement（V5.3）**
```
Operator (ROLE_PAYMASTER_SUPER) ─── SP.settleX402PaymentDirect(from, to, asset, amount, ...)
                                    ├─ 校验 nonce 未用 (x402SettlementNonces)
                                    ├─ asset.transferFrom(from, to, amount-fee)
                                    ├─ asset.transferFrom(from, SP, fee)
                                    └─ facilitatorEarnings[operator][asset] += fee
                                       (operator 后续可 withdrawFacilitatorEarnings)
```

**C. MicroPaymentChannel 生命周期**
```
Payer.openChannel(payee, token, deposit, salt)
  └─ token.transferFrom(payer, MPC, deposit) → channels[id].deposit

Payee.settleChannel(id, cumulativeAmount, sig)
  ├─ 校验 sig (payer 或 authorizedSigner) 通过 EIP-712 voucher
  ├─ delta = cumulative - settled
  └─ token.transfer(payee, delta) + settled = cumulative

Cooperative close: Payee.closeChannel(id, finalCumulative, sig)
  ├─ 校验 sig；finalize=true
  └─ token.transfer(payee, delta) + token.transfer(payer, deposit-cumulative)

Unilateral close: Payer.requestCloseChannel(id) → 等 900s → Payer.withdrawChannel(id)
  └─ Payee 在 900s 窗口内仍可 settleChannel；超时后 token.transfer(payer, deposit-settled)
```

### 关键架构发现（先于具体 finding）

> 标准 BLS 聚合签名（Boneh–Lynn–Shacham, 2003）的安全假设：合约**自己**根据 `signerMask` 从已注册公钥集合 `{pk_i}` 重建聚合公钥 `pk_agg = Σ pk_i`，然后 `e(G1, sig) == e(pk_agg, msgG2)` 才有意义。我们的实现把 `pk_agg` 作为 `proof` 参数让调用者提供 → **退化成单签**：调用者用任意私钥都能伪造任意 message 的签名。
>
> 单独修这一点不够 —`isValidator` 与 `blsPublicKeys` 的注入是 `onlyOwner`，**完全脱离** `Registry.ROLE_DVT` 的 30e GToken 锁仓门槛（即用户原本设计中的"经济保证"）。即便修复 PK 聚合，恶意/被盗 owner 仍可注入零质押的 dummy validator 操控共识。
>
> 再叠加 slash 的循环依赖（slash 恶意 validator 必须经过 BLS 共识，而 BLS 共识本身可能已被恶意 validator 控制）→ 治理路径无法自救。这就是把原 `B6-C1` 拆为 `C1a/C1b/C1c` 三独立 finding 的原因，每个对应一个独立的设计变更。

### 🔴 CRITICAL 严重度

#### B6-C1a：BLS 聚合签名 — 公钥未与已注册验证器集合（signerMask）交叉校验

- **位置**：
  - `BLSValidator.sol:26–48`（`verifyProof`）
  - `BLSAggregator.sol:243–284`（`_checkSignatures`）
  - `Registry.sol:344–361`（`batchUpdateGlobalReputation` 内联 BLS 校验段）
- **现象**：三处 BLS 验证均执行同一模式：
  ```
  proof = abi.encode(pkG1Bytes, sigG2Bytes, msgG2Bytes, signerMask)
  pk    = abi.decode(pkG1Bytes, ...)        ← 完全由调用者提供
  count = popcount(signerMask)               ← 仅校验数量 ≥ threshold
  pairing( e(G1_GEN, sig) == e(pk, msgG2) ) ← pk 无任何来源校验
  ```
  `BLSAggregator.blsPublicKeys[validator]` 在 `registerBLSPublicKey`（line 118–122）写入后，**全合约从未读取**（grep 已确认）。`signerMask` 从未与 `blsPublicKeys` 进行任何形式的聚合 PK 重建。
- **攻击路径**（仅需 EOA，**无门槛**，前提是已有合法 proposal 进入执行队列 — 见 B6-H1）：
  1. 攻击者本地生成新 BLS 密钥对 `(sk, pk = sk·G1)`。
  2. 计算 `expectedMessageHash`（按目标合约公开的消息构造规则）。
  3. 计算 `msgG2 = hashToG2(expectedMessageHash)`、`sig = sk · msgG2`。
  4. 任意构造 `signerMask`（只需 popcount ≥ `defaultThreshold`，例如 `0x07` 即 3 个 1）。
  5. 编码 `proof = abi.encode(pk, sig, msgG2, signerMask)`，调用对应入口。
  - 配对方程 `e(G, sk·msgG2) == e(sk·G, msgG2)` **数学上恒成立** → 验证通过。
- **直接危害**：通过 `executeWithProof` / `executeProposal` / `Registry.batchUpdateGlobalReputation` / `Registry.updateOperatorBlacklist` 任一入口，伪造 proof 即可触发：
  - **slash 任意 operator**（路径：`_executeSlash → SuperPaymaster.executeSlashWithBLS → GTokenStaking.slashByDVT`，可调高 `slashLevel` 直至 `MAX_SLASH_BPS` = 全额）。
  - **任意 user reputation 操控**（受 `_clampReputation` 100 BPS/epoch 限制，但多次 epoch 重放即可走完任意区间）。
  - **`executeProposal` 通用治理伪造**（`target.call(callData)` 可调任意合约的任意函数）。
- **Severity**：业务影响 = 5（协议级共识破坏）；可行性 = 4（数学可行但需配合 H1/H2 入口或合法 proposalId）；**总分 = 20 → CRITICAL**。
- **修复建议**（必须，与 C1b/C1c 联动）：
  - **方案 A — 标准 BLS 聚合**：`BLSAggregator._checkSignatures` 移除 `pkG1Bytes` 参数，改为根据 `signerMask` 从 `blsPublicKeys` + `validatorAt(uint256)`（新增有序索引）重建聚合 PK：
    ```solidity
    BLS.G1Point memory pkAgg = _zeroG1();
    for (uint256 i = 0; i < validatorCount; i++) {
        if (signerMask & (1 << i) != 0) {
            address v = validatorAt(i);
            pkAgg = BLS.add(pkAgg, _decodePK(blsPublicKeys[v].publicKey));
        }
    }
    require(pairing(G1_GEN, sig, pkAgg.neg(), msgG2));
    ```
  - 新 `proof` 编码：`abi.encode(sigG2Bytes, msgG2Bytes, signerMask)`（PK 字段彻底删除）。
  - **删除 `BLSValidator.sol`**：该合约的设计前提（caller 提供 PK）就是漏洞本身，工具职责完全可由 `BLSAggregator.verifyAggregated(signerMask, sig, msgG2)` 接口承担；保留 `utils/BLS.sol` 作为纯密码学库。
  - **Registry 内联校验路径调整**：`batchUpdateGlobalReputation` + `updateOperatorBlacklist` 直接 `IBLSAggregator(blsAggregator).verifyAggregated(...)`，不再做内联 pairing。Gas 影响：单次 staticcall ~2k overhead，相比 pairing precompile ~400k 几乎可忽略；高频场景下若需优化，可在 Registry 内 cache `blsAggregator` 的 `validatorCount` 与已注册 PK 的 hash（但短期不必要）。
- **回归测试**：
  - `test_BLS_RejectsForgedPK`：fresh sk 生成的合法 sig，proof 用 fresh pk → 预期 revert（`InvalidSignature`）。
  - `test_BLS_RejectsSignerMaskOutOfRange`：`signerMask` 包含 `validatorCount` 之外的 bit → revert。
  - `test_BLS_RebuiltPKMatchesOnchain`：模拟 4 个真 validator 注册 PK，链下聚合 → 链上 verifyAggregated → success。
  - `test_BLSValidator_Removed`：合约删除后 `Registry.blsValidator()` 应不再被引用。

#### B6-C1b：DVT 验证器集与 Registry stake-gate 完全解耦

- **位置**：
  - `DVTValidator.sol:55–57`（`addValidator` onlyOwner，无 stake 检查）
  - `BLSAggregator.sol:118–122`（`registerBLSPublicKey` onlyOwner，无 stake 检查）
- **现象**：两处 setter 均 `onlyOwner` 直接写 `isValidator[v] = true` / `blsPublicKeys[v] = pk`，**完全不读取** `Registry.hasRole(ROLE_DVT, v)` 或 `GTokenStaking.roleLocks[v][ROLE_DVT].amount`。设计文档暗示"validator 必须 ROLE_DVT + 锁仓 ≥ 30e GToken"，但合约层从未强制。
- **攻击路径 / 失效模式**：
  1. 恶意/被盗 owner（或 1-of-N 多签被攻破）调用 `addValidator(attacker)` + `registerBLSPublicKey(attacker, fakePK)`。
  2. attacker 0 资金即获得 BLS 共识权 → 配合 C1a 修复后仍可正常签名走完所有 BLS 路径。
  3. slash 失败保护（attacker 无 GToken 可被 slash）→ Tier-2 governance slash 也无法没收资金。
- **业务影响**：5（共识权与经济保证脱钩，slash 罚没机制成为空话）；可行性 = 4（需要 owner 操作；但 owner 本身就是单点信任）；**总分 = 20 → CRITICAL**。
- **修复建议**：
  - 在 `DVTValidator.addValidator` 内强制：
    ```solidity
    function addValidator(address v) external onlyOwner {
        bytes32 ROLE_DVT = keccak256("DVT");
        require(IRegistry(REGISTRY).hasRole(ROLE_DVT, v), "NotDVTRole");
        uint256 minStake = IRegistry(REGISTRY).getRoleConfig(ROLE_DVT).minStake;
        uint256 locked = IGTokenStaking(STAKING).roleLocks(v, ROLE_DVT).amount;
        require(locked >= minStake, "InsufficientLockedStake");
        isValidator[v] = true;
        emit ValidatorAdded(v, locked);
    }
    ```
  - 在 `BLSAggregator.registerBLSPublicKey` 内做同样的 stake-gate（避免 PK 注入路径绕过）。
  - 新增 `removeValidator` + `unregisterBLSPublicKey`（与 B6-M3 联动）：当 `Registry` 通知 ROLE_DVT 被退出/slash 至阈值以下时同步清理。
  - 不要 hardcode `30 ether`；动态读取 `RoleConfig.minStake`，与 Registry 内 `_initRole(ROLE_DVT, 30 ether, 3 ether, ...)` 保持一致。
- **回归测试**：
  - `test_AddValidator_RevertsIfNotDVTRole`。
  - `test_AddValidator_RevertsIfStakeBelowMinStake`。
  - `test_AddValidator_SucceedsAtExactMinStake`。
  - `test_AddValidator_PicksUpRoleConfigChange`：DAO 把 `minStake` 上调后，新 validator 必须满足新阈值。

#### B6-C1c：Slash 循环依赖 — 罚没恶意 validator 必须经过同一套 BLS 共识

- **位置**：架构层面问题（涉及 `GTokenStaking.slashByDVT`、`SuperPaymaster.executeSlashWithBLS`、`BLSAggregator._executeSlash`）
- **现象**：当前 slash 仅有一条路径 — `BLSAggregator.verifyAndExecute → _executeSlash → SP / GTokenStaking.slashByDVT`。如果 BLS 共识被攻破（C1a + C1b 任一），slash 恶意 validator 必须**通过被攻破的同一条链路**，链路本身已被攻击者控制。
- **业务影响**：4（治理瘫痪；恶意 validator 无法被罚没，需要 owner 紧急升级合约）；可行性 = 3（需配合 C1a / C1b 已经发生）；**总分 = 12 → HIGH**。但因其属于"系统性可恢复性丧失"，按用户决策（方案 A）将其纳入 CRITICAL 必修清单。
- **修复建议**（设计已在独立文档定稿，待 Phase 5 实施）：
  - 引入**独立 governance slash 路径**（不依赖 BLS 共识）— 详见 `docs/security/2026-04-25-governance-slash-design.md`。
  - 核心设计（Optimistic Governance）：
    1. **Stage 1 Propose**：DAO 多签（Safe Wallet 离线 1 次签字）调用 `GovernanceSlasher.propose(target, slashLevel, reason, bond)`。
    2. **Stage 2 Challenge Window**：72h，permissionless — 任何人可质押 bond 提出挑战，进入仲裁流程。
    3. **Stage 3 Execute**：72h 后无未决挑战，permissionless EOA 调用 `execute(proposalId)`，直接调用 `GTokenStaking.slashByGovernance(target, level)` —— 不经过 BLS 路径。
    4. **Stage 4 FastTrack**（可选，更高多签门槛）：跳过 challenge window，应对极端紧急情况。
  - `GTokenStaking` 新增 `slashByGovernance(address user, uint256 slashBps)`，授权 modifier 改为 `onlyAuthorizedSlasher` + `address.code.length > 0` 双校验，确保是 GovernanceSlasher 合约本体。
  - 关键安全点：multisig 私钥**永不在线**（Safe Wallet UI 离线聚合）；execute 步骤是 permissionless；challenge window 由全社区监督。
- **回归测试**（详见设计文档 §7）：
  - `test_GovSlash_HappyPath_AfterChallengeWindow`。
  - `test_GovSlash_RejectsExecuteBeforeWindow`。
  - `test_GovSlash_ChallengeBondReturnsOnValidProposal`。
  - `test_GovSlash_BypassesBLSEvenIfBLSAggregatorIsRogue`。
  - `test_GovSlash_FastTrackRequiresHigherThreshold`。

#### B6-C2（原 H2 升级）：`Registry.updateOperatorBlacklist` 允许调用者跳过 BLS 校验

- **位置**：`Registry.sol:377–393`
- **现象**：
  ```solidity
  if (address(blsValidator) != address(0) && proof.length > 0) {
      // BLS check
  }
  ISuperPaymaster(SUPER_PAYMASTER).updateBlockedStatus(operator, users, statuses);
  ```
  调用者只需传 `proof = ""` 即可跳过 BLS 验证；前提是 `isReputationSource[msg.sender]` 通过。
- **业务影响**：任意 reputation source（包括 BLSAggregator、ReputationSystem）皆可不经任何 DVT 共识，将任意 user 加入/移除任意 operator 的 blacklist。结合 C1a，攻击者可：
  1. 任意调用 `BLSAggregator.executeProposal(target=Registry, callData=updateOperatorBlacklist(...))`，且 callData 内的 `proof` 字段可设为空 → Registry 完全跳过 BLS。
  2. 直接通过未来的任意 wrapper（若注册为 reputation source）。
- **Severity**：业务影响 = 4；可行性 = 4（仅需被设为 reputation source 的合约）；**总分 = 16 → CRITICAL**（按 rubric 严格归类）。
- **修复建议**：
  - 强制 `if (proof.length == 0) revert BLSProofRequired();`，与 `batchUpdateGlobalReputation` 保持一致。
  - 与 C1a 修复联动：改为 `IBLSAggregator(blsAggregator).verifyAggregated(...)` 接口。
  - 同时检查 `_msgSender` 是否为 `BLS_AGGREGATOR_ADDRESS`（白名单），单一可信入口。
- **回归测试**：
  - `test_UpdateBlacklist_RevertsOnEmptyProof`。
  - `test_UpdateBlacklist_RevertsIfCallerNotBLSAggregator`。
  - `test_UpdateBlacklist_HappyPathViaBLS`。

### 🔴 HIGH 严重度

#### B6-H1：`DVTValidator.executeWithProof` 无调用者鉴权

- **位置**：`DVTValidator.sol:86–105`
- **现象**：函数 `external` 无任何 modifier 或 `if (msg.sender ...)` 检查；它仅检查 `proposals[id].executed == false`。配合 C1a 即可"任意人调用 → 任意 slash + 任意 reputation"。
- **既存补偿**：`createProposal`（line 59）有 `isValidator` 检查，所以 `proposalId` 必须是已存在 proposal — 但 `nextProposalId` 从 1 顺序递增，且 `proposals[id]` 的 `operator/slashLevel/reason` 在 proposal 创建后无法修改，所以攻击者只能 slash **已被验证器创建过 slash proposal 的 operator**（仍是严重危害）。
- **业务影响**：4（slash 抢跑、消息绑定操控）；可行性 = 5（无门槛 EOA）；**总分 = 20 → 修复 C1a 之后降为 8 (MEDIUM)**；本 finding 严重度评级 = 12 (HIGH)（在 C1a 已计 CRITICAL 的前提下，此项再单独按 C1a 后状态评级）。
- **修复建议**：
  - 短期：限制 `executeWithProof` 仅 `BLS_AGGREGATOR` 或 `isValidator[msg.sender]` 调用（防止抢跑 + spam）。
  - 长期：与 C1a 联动，PK 来源校验做对了之后，本接口的"无鉴权"在数学上不再可被利用 — 但仍建议保留 `onlyValidator` 防止恶意 spam。
- **回归测试**：`test_ExecuteWithProof_RevertsForAnonymousCaller`。

### 🟡 MEDIUM 严重度

#### B6-M1：BLSAggregator 与 Registry 消息哈希不一致 — slash + reputation 同 proof 必反

- **位置**：
  - BLSAggregator: `expectedMessageHash = keccak256(abi.encode(proposalId, operator, slashLevel, repUsers, newScores, epoch, chainid))`（line 143–151）
  - Registry: `messageHash = keccak256(abi.encode(proposalId, address(0), uint8(0), users, newScores, epoch, chainid))`（line 357–360）
- **现象**：当 `verifyAndExecute` 同时携带 `operator != 0` 且 `repUsers.length > 0` 时，BLSAggregator 检查使用真 operator/slashLevel，Registry 检查则强制用 `0/0` — 两个 hash 不可能同时匹配同一份 proof。
- **业务影响**：3；可行性 = 4；总分 = 12 → MEDIUM。功能性 bug，迫使 slash + reputation 必须分两次提案，易导致集成方误用。
- **修复建议**：让 Registry 的 `messageHash` 复用 BLSAggregator 传入的语义（增加 `operator/slashLevel` 参数），或在 Registry 暴露 `batchUpdateGlobalReputationWithSlashContext(operator, slashLevel, ...)` 由 BLSAggregator 调用。

#### B6-M2：`BLSAggregator` 阈值 setter 无 timelock 与下限保护

- **位置**：line 86–98（`updateThreshold`、`updateMinThreshold`）
- **现象**：owner 一笔交易即可将 `defaultThreshold` 改为 1（已 by setter 保证 ≥ minThreshold；但 `updateMinThreshold` 同时支持下调 minThreshold，最低不见下限 — 仅 `_threshold > 0` 与 `<= MAX_VALIDATORS`）。
- **业务影响**：3；可行性 = 3；总分 = 9 → MEDIUM。即便 C1a 修复后，仍可大幅削弱共识强度。
- **修复建议**：(a) `minThreshold` 增设硬下限（建议 ⌈validatorCount/3⌉ 或 3，取大者）；(b) 引入 timelock（24–72h）；(c) 完整事件审计。

#### B6-M3：`DVTValidator` 验证者集只增不减 — `addValidator` 无对应 `removeValidator`

- **位置**：line 55–57
- **现象**：`isValidator[_v] = true;` — 没有删除路径，验证器一旦添加即永久。被盗或离职的验证器无法移除。
- **业务影响**：2；可行性 = 3；总分 = 6 → MEDIUM。结合 C1b 修复后必须配套 `removeValidator`（否则 stake-gate 检查只在 add 时生效，后续 unstake 无法移除）。
- **修复建议**：新增 `removeValidator(address)` onlyOwner，并在 BLSAggregator 侧 `unregisterBLSPublicKey`；与 Registry 退出路径联动（Registry exit ROLE_DVT 时回调 DVTValidator）。

#### B6-M4：`Registry.batchUpdateGlobalReputation` threshold fallback 静默 hardcode = 3

- **位置**：`Registry.sol:347–351`
- **现象**：
  ```solidity
  uint256 threshold = 3;
  if (blsAggregator != address(0)) {
      try IBLSAggregator(blsAggregator).defaultThreshold() returns (uint256 t) {
          threshold = t;
      } catch {}
  }
  ```
  若 `blsAggregator == address(0)` 或调用 revert，则静默使用 `threshold = 3`。
- **业务影响**：3；可行性 = 3；总分 = 9 → MEDIUM。在初始化未完成、blsAggregator 被卸载、或合约升级期间出现 ABI 不兼容时，threshold 自动降为 3，无任何告警。
- **修复建议**：
  - `blsAggregator == address(0)` 时直接 revert（强制配置）。
  - try/catch 失败时 revert，而不是 fallback 到任意常量。

#### B6-M5：`ReputationSystem.syncToRegistry` 任意 EOA 可调用，proposalId 由 caller 控制

- **位置**：line 168–187
- **现象**：函数 `external` 无鉴权；`proposalId = keccak256(user, epoch)` 完全由 caller 决定。`epoch` 由 caller 提供，无单调递增校验（`Registry` 在 `lastReputationEpoch[user]` 处仅 skip 不 revert）。
- **业务影响**：3；可行性 = 4；总分 = 12 → MEDIUM。攻击者可：
  - 预构造任意 (user, epoch) → 唯一 proposalId，提交伪造 BLS proof（依赖 C1a）即可写入 reputation。
  - 即使 C1a 修复，仍可 DoS 真实 sync：抢跑相同 (user, epoch) → `executedProposals[proposalId] = true` → 真实验证器无法再用同一 proposalId 提交。
- **修复建议**：
  - 增加 `isAuthorizedSyncer[msg.sender]` 鉴权，仅授权 keeper / DVT 节点可调（与 dvt-proof-scenarios 设计的 Reputation Epoch 场景对齐）。
  - 或在 ReputationSystem 内强制 `epoch > lastSyncEpoch[user]` 以防 epoch 重放。

### 🟢 LOW 严重度（择重列出）

- **B6-N1**：`BLSValidator.verifyProof` 解码 `signerMask` 但完全未使用（line 26）— 与 C1a 一并通过删除 BLSValidator 解决。

- **B6-N2**：`BLSAggregator.executeProposal` 通用治理无 `target` 白名单（line 187–228）— 配合 C1a 即任意合约任意调用。即使修复 C1a，仍建议增加 `allowedTargets[address] => bool` 白名单（Registry / SuperPaymaster / GTokenStaking 三个核心 target），不允许自由 target.call。

- **B6-N3**：`BLSAggregator` 构造函数允许 `_superPaymaster == address(0)` 与 `_dvtValidator == address(0)`（line 103–112）— 仅 `_registry` 做 0 检查；后续 `_executeSlash` 路径若 SP=0 会 revert，但 silent 部署可能造成线上误配置。建议三者皆校验。

- **B6-N4**：`DVTValidator.setBLSAggregator`（line 107–109）和 `BLSAggregator` 内多个 setter 缺事件 — 治理可见性差。

### 🔴 HIGH 严重度

#### P3-H1：`PaymasterBase.setCachedPrice` Owner 任意价格写入，**完全跳过 MIN/MAX 边界检查**

- **位置**：`contracts/src/paymasters/v4/PaymasterBase.sol:474–478`
  ```solidity
  function setCachedPrice(uint256 price, uint48 timestamp) external onlyOwner {
      if (price == 0) revert Paymaster__InvalidOraclePrice();
      cachedPrice = PriceCache({ price: uint208(price), updatedAt: timestamp });
      emit PriceUpdated(price, timestamp);
  }
  ```
- **现象**：`updatePrice()`（line 462–470）走 Chainlink 路径时严格检查 `MIN_ETH_USD_PRICE < price < MAX_ETH_USD_PRICE`；但 `setCachedPrice` 仅检查 `price != 0`。Owner 可设任意 `uint208`（1 至 ~4.1e62）作为 ETH/USD 价格，引发 paymaster 全量 token 计算错乱。
- **业务影响**：4（V4 AOA paymaster operator 的 owner key 被盗即可瞬间清空合约 deposit；用户被多扣或 paymaster 资不抵债）；可行性 = 3（owner key 单点）；**总分 = 12 → HIGH**。
- **修复建议**：
  - 在 `setCachedPrice` 内加同样的 `MIN_ETH_USD_PRICE / MAX_ETH_USD_PRICE` 检查。
  - 可选：增加 timelock + multisig 治理（与 V3 SP 的 owner override 一致性）。
  - 增加专属事件 `OwnerPriceOverride(price, timestamp, reason)` 区别于 keeper-pushed `PriceUpdated`。
- **回归测试**：
  - `test_SetCachedPrice_RevertsBelowMin` / `test_SetCachedPrice_RevertsAboveMax`。
  - `test_SetCachedPrice_RejectedByNonOwner`。

#### P3-H2：`SuperPaymaster.updatePriceDVT` Chainlink down 时 owner 任意价格通过

- **位置**：`SuperPaymaster.sol:342–376`
- **现象**：`updatePriceDVT(price, updatedAt, proof)` allow `msg.sender == owner()` 作为紧急通道。Line 361–376 通过 try/catch 调 `latestRoundData()` 做 ±20% deviation 检查；如果 Chainlink revert（line 373 `catch`），**deviation 检查被跳过**，owner 的任意 price（仍受 MIN/MAX bounds 约束，line 357）被无差别接受。
- **业务影响**：4（owner 可在 Chainlink 短暂 outage 时操控价格；MIN/MAX 边界过宽 — 例如 100x ETH 实际价仍在 bounds 内）；可行性 = 3（需 Chainlink down 或 outage）；**总分 = 12 → HIGH**。
- **修复建议**：
  - Chainlink down 时拒绝 owner 路径（`catch { if (msg.sender != BLS_AGGREGATOR) revert OracleError(); }`），强制走 BLS 路径或社区 governance fast-track。
  - 收紧 MIN/MAX bounds（建议 `MIN_ETH_USD_PRICE = 100e8`、`MAX_ETH_USD_PRICE = 100_000e8`，或动态相对于 last cached price ±50%）。
  - 增加事件 `OwnerEmergencyPriceUpdate(price, chainlinkAvailable)`。
- **回归测试**：
  - `test_DVTUpdate_RevertsOwnerPathWhenChainlinkDown`。
  - `test_DVTUpdate_RejectsExcessiveDeviationWhenChainlinkUp`。

#### P3-H3：Chainlink `answeredInRound >= roundId` 校验缺失

- **位置**：
  - `SuperPaymaster.sol:656–680`（`updatePrice`）
  - `SuperPaymaster.sol:688`（`_calculateAPNTsAmount` realtime path）
  - `PaymasterBase.sol:352, 463`（同两处）
- **现象**：`latestRoundData()` 返回 5 元组 `(roundId, price, startedAt, updatedAt, answeredInRound)`，所有调用点丢弃了 `answeredInRound`（用 `,` 占位）。Chainlink best practice 要求 `answeredInRound >= roundId`，否则可能拿到未完成 round 的数据。
- **业务影响**：3（在 Chainlink 出现 stale round bug 时引发错价；`updatedAt < threshold` 检查 partial 覆盖）；可行性 = 4（依赖 Chainlink 异常事件，被动触发）；**总分 = 12 → HIGH**。
- **修复建议**：
  ```solidity
  (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) = ETH_USD_PRICE_FEED.latestRoundData();
  if (answeredInRound < roundId) revert OracleError();
  ```
- **回归测试**：在 `MockChainlink` 中模拟 `answeredInRound < roundId` → 预期 revert。

#### P3-H4：`postOp` `recordDebt` fallback 累积无上限债务，`retryPendingDebt` 仅手动可调

- **位置**：`SuperPaymaster.sol:868–875`（postOp 内 try/catch + recordDebt 调用）+ `xPNTsToken.sol:330–343`
- **现象**：postOp 主路径 `IxPNTsToken(token).burnFromWithOpHash(user, finalXPNTsDebt, userOpHash)` 失败（用户 xPNTs 余额不足等）时 fallback 到 `recordDebt(user, finalXPNTsDebt, userOpHash)` 累积成 pending debt。由 `retryPendingDebt(user, opHash)`（line 919–925）由 keeper 手动重试。**没有自动清理或债务利息机制**，如果用户永不补充 xPNTs 余额，debt 将无限累积。同时 operator 已被扣 aPNTs，相当于 operator 替用户付了 gas，需自行追讨 — 这是经济激励问题。
- **业务影响**：3（operator 的 aPNTs 与 user xPNTs debt 之间存在跨账户应收风险）；可行性 = 4（被动累积，正常使用就可能触发）；**总分 = 12 → HIGH**。
- **修复建议**：
  - 给 pendingDebts 加 TTL：超过 N 天未补足 → operator 可标记坏账并向 user 永久 blacklist（`updateBlockedStatus`）。
  - 引入"debt cap per user"（例如 ≥ $50 等值后阻止新 sponsorship）。
  - 增加事件 `DebtAccumulated(user, amount, totalDebt)` 便于 SDK 监控。
- **回归测试**：
  - `test_PostOp_DebtRecordedWhenInsufficientBalance`。
  - `test_RetryDebt_SuccessAfterTopUp` / `test_RetryDebt_RevertsBeforeTopUp`。
  - `test_DebtCap_BlocksFurtherSponsorshipAfterCap`。

#### P3-M2：Owner override `setCachedPrice` 无紧急 pause 机制（双合约通病）

- **位置**：`SuperPaymaster.sol:342`（owner emergency path）+ `PaymasterBase.sol:474`（owner direct override）
- **现象**：两个合约的 owner key 一旦被盗，攻击者可瞬间设置任意（在 MIN/MAX 内的）价格并立即生效，**没有 pause 或 timelock 机制**。`OperatorConfig.isPaused` 仅暂停 sponsorship，不影响 owner 价格写入。
- **业务影响**：3；可行性 = 3；总分 = 9 → MEDIUM。owner 单点是设计上接受的风险，但缺紧急 pause 让事故影响放大。
- **修复建议**：
  - 引入 `oraclePaused` flag（独立于 operator pause），由 owner 或社区多签触发，paused 时全部 price-derived 计算 revert。
  - 与 P3-H1/P3-H2 联动 — 收紧 MIN/MAX 边界 + pause 机制。

#### P3-M3：`x402SettlementNonces` 全局 mapping 无清理路径，长期增长

- **位置**：`SuperPaymaster.sol:952`（state）+ `1138–1139`（写入）
- **现象**：每次 `_validateX402AndComputeFee` 写一条 `nonces[bytes32] = true`。Mapping 无 expiry，永久累积。USDC 自身有 EIP-3009 nonce 机制，所以**这层重复 tracking 是冗余**。
- **业务影响**：2（链状态膨胀，gas 费成本随时间上升）；可行性 = 3（被动累积）；**总分 = 6 → MEDIUM**。
- **修复建议**：
  - **方案 A**：删除 SP 内 `x402SettlementNonces` 检查，完全依赖底层 token（USDC EIP-3009）的 nonce 检查。
  - **方案 B**：保留并加入 expiry — `mapping(bytes32 => uint256) nonceUsedAt`，如果 `block.timestamp - nonceUsedAt[nonce] > 7 days` 允许清理（清理操作可由 keeper 调）。

#### P3-M4：`PaymasterBase.setCachedPrice` 与 `updatePrice` 之间 cachedPrice.decimals 不一致

- **位置**：`PaymasterBase.sol`（PriceCache struct + `setCachedPrice` line 474–478）
- **现象**：`PriceCache` 在 PaymasterBase 中是 `{ price: uint208, updatedAt: uint48 }`（无 decimals 字段）；decimals 由全局 `oracleDecimals` 跟踪。如果 owner 在升级 / 切换 oracle 后调 `setCachedPrice`，但忘记同步更新 `oracleDecimals`，decimals 不匹配会让 `_calculateTokenCost` 静默错算。
- **业务影响**：3；可行性 = 2（需配置错误）；**总分 = 6 → MEDIUM**。
- **修复建议**：将 decimals 纳入 PriceCache struct（PaymasterBase 与 SuperPaymaster 一致），或在 `setCachedPrice` 强制 caller 同时传入 decimals 并校验。

#### P3-M5：`xPNTs` ERC20Permit nonce 抢跑（标准 EIP-2612 边角，`burnFromWithOpHash` 间接缓解）

- **位置**：`xPNTsToken.sol`（继承 OZ ERC20Permit）+ `usedOpHashes` mapping（line 305 等）
- **现象**：xPNTsToken 继承 OZ ERC20Permit，每用户单调 nonce — 用户同时签两份 permit 时，先上链的成功，后上链的失败。这是 EIP-2612 标准问题。Paymaster 的 `burnFromWithOpHash`（line 298–319）通过 `usedOpHashes[userOpHash]` 防 op 级 replay，缓解了 paymaster 内的 permit 抢跑（不直接调 permit，而是用 autoApprovedSpenders 走 burnFrom）。**SDK 直接调用 permit 的非 paymaster 场景仍有抢跑风险**。
- **业务影响**：2；可行性 = 3；**总分 = 6 → MEDIUM**。
- **修复建议**：SDK 文档化 permit 抢跑模式，建议 deadline + retry。或在 xPNTsToken 添加 ERC2612 permit 的 fallback "permit 失败时不 revert，仅返回 false" — 不推荐，破坏标准兼容。

- **位置**：`SuperPaymaster.sol:656–665`（`updatePrice`）

- **业务影响**：1；可行性 = 1；总分 = 1 → LOW（隐式正确，仅可读性）。
- **修复建议**：显式加 `if (updatedAt == 0) revert` 与 V4 一致。

### J1：社区注册 → operator 上线

**步骤链**：
1. `Registry.registerRole(ROLE_COMMUNITY, self, communityRoleData)` — 锁仓 GToken（如有 minStake）+ ticket fee
2. `Registry.registerRole(ROLE_PAYMASTER_SUPER, self, paymasterRoleData)`（前置依赖：`hasRole[ROLE_COMMUNITY]`）
3. `xPNTsFactory.createXPNTs(self, name, symbol, ...)` 部署专属 xPNTs clone
4. `SuperPaymaster.configureOperator(xPNTsToken, treasury, exchangeRate)`
5. `SuperPaymaster.deposit(amount)` 充值 aPNTs
6. （可选）`setOperatorLimits(minTxInterval)`

**断点 / Major UX 问题**：

- **J1-MAJOR-1**：`configureOperator`（SP:215–245）的错误码 `InvalidConfiguration` 在 (xPNTsToken, treasury, exchangeRate) 三个字段任一为零时统一抛出，**operator 无法判断哪个字段错**。
  - 修复建议：拆成 `InvalidXPNTsToken` / `InvalidTreasury` / `InvalidExchangeRate` 分别带字段名。

- **J1-MAJOR-2**：`configureOperator` 不支持"只更新单一字段"。修改 treasury 必须重新提交全部三个字段，operator 容易在前端误改 exchangeRate。
  - 修复建议：拆出 `setOperatorTreasury` / `setOperatorExchangeRate` / `setOperatorXPNTsToken` 三个独立 setter，与现有 `setOperatorLimits` 风格一致。

- **J1-MINOR-1**：`xPNTsFactory.getTokenAddress(operator)` 是关键依赖，但 `configureOperator` 失败时只抛 `InvalidXPNTsToken` / `FactoryVerificationFailed` — 没说明"operator 必须先调 factory.createXPNTs"。
  - 修复建议：`FactoryVerificationFailed` 错误信息文档化，或在 JSDoc 中给出 step-by-step。

**恢复缺失**：
- 如果 operator 注册了 ROLE_PAYMASTER_SUPER 但忘了调 `configureOperator`，validate 会 silent reject（J2 描述的问题）— operator 自己难以诊断。
  - 修复建议：增加 `getOperatorReadiness(operator)` view function，返回 (isRegistered, isCommunity, isConfigured, isPaused, hasBalance, isReady) tuple，供 SDK 健康检查。

### J2：用户领 SBT → 首次 sponsored userOp

**步骤链**：
1. 用户有 AA 钱包（AirAccount/SimpleAccount）
2. `Registry.registerRole(ROLE_ENDUSER, user, enduserRoleData)` → mint MySBT + Registry 调 `SP.updateSBTStatus(user, true)`
3. 用户构造 userOp，paymasterAndData = [paymaster(20) ‖ paymasterValidationGasLimit(16) ‖ paymasterPostOpGasLimit(16) ‖ operator(20) ‖ maxRate(32)]
4. EntryPoint 调 `SP.validatePaymasterUserOp` → 走完 5 个 silent failure 检查
5. 用户操作执行
6. EntryPoint 调 `SP.postOp` → operator 扣 aPNTs，用户扣 xPNTs（或 fallback 到 recordDebt）

**断点 / Blocker / Major UX 问题**：

- **J2-BLOCKER-1**：`validatePaymasterUserOp` 5 个失败分支（line 738/748/753/761/783/795）**全部 silent return** `_packValidationData(true, 0, 0)`，**不带任何 reason**：
  ```
  - !isConfigured                → silent reject
  - isPaused                     → silent reject
  - !isEligibleForSponsorship    → silent reject
  - userState.isBlocked          → silent reject
  - exchangeRate > maxRate       → silent reject
  - aPNTsBalance < required      → silent reject
  ```
  bundler 拿到通用 "AA34 paymaster validation failed"，前端无法区分原因。**这是产品层最严重的 UX 问题**。
  - 根因：V3.6 fix 为了避免 banned opcode revert，把所有失败路径改成 silent — 但 silent reject 与 revert 不可兼得，至少应该提供一个 off-chain 可查的原因。
  - 修复建议：
    - **方案 A**：增加 `view function dryRunValidation(userOp, maxCost) returns (bool, uint8 reasonCode, string reason)`，前端先调它再正式提交。
    - **方案 B**：在 `validatePaymasterUserOp` 前给前端的 SDK 函数中预跑所有检查并返回结构化错误。
    - **方案 C**（最小改动）：每个失败分支前 emit 一条 event（虽然 validation 阶段不能 emit，但 view function 可以做镜像检查并 emit 在 dryRun 路径中）。

- **J2-MAJOR-3**：`validUntil = uint48(cachedPrice.updatedAt + priceStalenessThreshold)`（line 743）— 如果 cachedPrice 已过期，`validUntil` 在过去 → EntryPoint 返回 `0xb84c4c70` (PaymasterValidationData expired)，**与 paymaster 自身错误混淆**，前端难诊断"是 oracle 失效还是 user 没 SBT"。
  - 修复建议：在 SDK 暴露 `getOracleStatus()` 和 `getCachedPriceAge()` 让前端先检查。

- **J2-MAJOR-4**：`isEligibleForSponsorship(user)` 逻辑 = `sbtHolders[user] || isRegisteredAgent(user)`。如果 `sbtHolders[user]` 状态与 `MYSBT.userToSBT[user]` 不同步（INV-02 已被 Phase 1 视为风险），用户可能持有 SBT 但 SP 端 `sbtHolders=false`，sponsorship 被拒，**用户无法自助修复**（需 operator 或 admin 调用 Registry 同步）。
  - 修复建议：增加 `Registry.resyncSBTStatus(user)` 公开 view + 任意人可触发的同步函数。

- **J2-MINOR-2**：用户首次 sponsored userOp 期间 `xPNTs` 余额不足，进入 `recordDebt` 路径（P3-H4）— 用户体验是"看似成功"但实际产生债务，**前端没有"你欠 X xPNTs"的提示界面**。
  - 修复建议：postOp emit `DebtIncurred(user, opHash, amount)` 事件（已有），SDK 必须 subscribe 并主动提示用户。

**恢复缺失**：
- 用户如果 ENDUSER 注册时 `roleData` 编码错误（abi 解码失败），整个 tx revert，但用户已支付 ticket fee 吗？需查 Registry `_validateAndProcessRole` 是否做 atomic rollback。代码层 atomic 的（同一 tx），但前端如果只 catch 通用错就给用户"重试"按钮，他可能多花 gas。
  - 修复建议：SDK 提供 `validateRoleData(roleId, data)` view 函数，提交前先校验。

### J3：Operator 充值 aPNTs → 出价 → 提取 protocolRevenue

**步骤链**：
1. Operator 通过未来的 mint 路径或市场获得 aPNTs（**当前 aPNTs token 的 mint 权限模型未文档化**）
2. `aPNTs.approve(SuperPaymaster, amount)`
3. `SuperPaymaster.deposit(amount)` → 转入 + `operators[op].aPNTsBalance += amount`
4. `SuperPaymaster.configureOperator(...)` 调整 exchangeRate（in-flight 影响）
5. （Owner 路径）`withdrawProtocolRevenue(treasury, amount)` 把累积 protocolRevenue 提走

**断点 / Major UX 问题**：

- **J3-MAJOR-5**：aPNTs token 的 mint 权限模型在文档与代码中**不清晰**。`SuperPaymaster.APNTS_TOKEN`（line 70）只是一个 ERC20 引用 — operator 如何获得 aPNTs？是协议 mint？市场购买？质押 GToken 兑换？目前代码没说明。
  - 修复建议：文档化 aPNTs 经济模型（建议在 `docs/economic-model.md`），或者在 `setAPNTsToken` 处加入 minter 角色注释。

- **J3-MINOR-3**：`deposit(amount)` 走 `transferFrom` 需要先 approve，但如果 operator 忘记 approve，错误来自底层 ERC20（`ERC20InsufficientAllowance`）— SP 层没有 user-friendly 提示。
  - 修复建议：SDK 在 `depositAPNTs` 前先检查 allowance；前端给 "Approve & Deposit" 两步按钮。

- **J3-MINOR-4**：`configureOperator` 调整 exchangeRate 是**立即生效**，没有 timelock。已被签名但未上链的 userOp 可能用旧 rate，已上链未 postOp 的不受影响（context 中存了 exchangeRate 在 opcontext 中），但 in-flight 期 rate change 会导致用户超扣或 operator 多扣。
  - 修复建议：增加 timelock（例如 5 min grace period）+ event `OperatorRateChangePending(operator, oldRate, newRate, effectiveAt)`。

- **J3-MINOR-5**：`withdrawProtocolRevenue` 是 owner-only。但 `protocolRevenue` 与 `operators[op].aPNTsBalance` 共享 `totalTrackedBalance`（INV-03）。owner 提走 protocolRevenue 不影响 operator 余额，但**没有 safety margin** — owner 错误地把 in-flight buffer 也提走会导致 postOp underflow（line 859–862 的 `ProtocolRevenueUnderflow` event 已经做了一部分防御）。
  - 修复建议：`withdrawProtocolRevenue` 增加"保留 buffer"参数：`withdrawProtocolRevenue(treasury, amount, minReserve)`。

### J4：x402 一次性支付（EIP-3009）

**步骤链**：
1. dApp/Agent 在 `paymasterAndData` 中嵌入 x402 payload
2. Operator 拿到 EIP-3009 签名 (from, to, asset, amount, validAfter, validBefore, nonce, signature)
3. Operator 调 `SP.settleX402Payment(...)` → SP 验 nonce + 算 fee + 调 USDC `transferWithAuthorization` + 转给 to

**断点 / Major UX 问题**：

- **J4-MAJOR-6**：客户端如何获得正确的 facilitator fee？逻辑是：
  ```
  effectiveFee = operatorFacilitatorFees[operator] != 0 
                 ? operatorFacilitatorFees[operator] 
                 : facilitatorFeeBPS
  ```
  客户端必须先调 `getEffectiveFacilitatorFee(operator)`（合约**没有这个 view**），目前需要 SDK 自己读两个 mapping 比对。
  - 修复建议：在 SP 增加 `getEffectiveFacilitatorFee(operator) view returns (uint256)` 一行 view 函数，SDK 直接消费。

- **J4-MINOR-6**：`settleX402Payment` 的 nonce 由 caller 提供，目前 SDK 用 keccak256(fingerprint) 还是 random？没文档化。
  - 修复建议：SDK 文档统一规范（建议 `keccak256(operator, asset, from, to, amount, timestamp_ms, random_salt)`，避免冲突也保证可观察性）。

- **J4-MINOR-7**：x402 settle 失败的常见原因：
  1. signature 错误 → USDC 内部 revert，错误信息来自 USDC 合约
  2. nonce 已用 → SP `NonceAlreadyUsed`
  3. 时间窗过期 → USDC 内部 revert
  4. allowance 不足（仅 Direct 路径）→ SafeERC20 revert
  
  各种原因抛出位置不一，SDK 难以聚合错误码。
  - 修复建议：SDK 包装 `tryX402Settle()` 函数把所有错误归一化为统一 enum（NonceUsed / SignatureInvalid / TimeExpired / Unauthorized / Insufficient / Other）。

**恢复缺失**：
- 失败 settle 没有事件日志（仅成功路径 emit `X402PaymentSettled`），调试 settlement 失败需要事后链上交易回放。
  - 修复建议：增加 `X402PaymentFailed(operator, asset, nonce, reason)` event 在 try/catch 失败路径中，但要小心 nonReentrant 与 fail-fast 的 trade-off。

### J5：MPC 通道开 / 补 / 关 + voucher 离线签名

**步骤链**：
1. Payer (Agent) 调 `MicroPaymentChannel.openChannel(payee, token, deposit, salt)` → 锁仓 token
2. 服务期间 payee 累积服务量；payer（或 authorizedSigner = AirAccount Session Key）签 voucher (channelId, cumulativeAmount)
3. payee 调 `settleChannel(channelId, cumulativeAmount, signature)` → 立刻提取 (cumulativeAmount - settled)
4. payee 调 `closeChannel(channelId, finalAmount, signature)` 或 payer 调 `requestCloseChannel(channelId)`
5. CLOSE_TIMEOUT (15 min) 后 payer 调 `withdrawChannel(channelId)` 取回剩余 deposit

**断点 / Major UX 问题**：

- **J5-MAJOR-7**：CLOSE_TIMEOUT = 15 min（line 35）— 对**实时服务**够，但对**异步服务**（Agent 离线 24h、跨时区结算）**严重不足**。如果 payee 离线超 15 min，payer 可发起 close 并 15 min 后强制 withdraw —— 已欠的服务费可能被赖账。
  - 修复建议：把 `CLOSE_TIMEOUT` 改为 `immutable` 在构造时配置（默认 15 min，企业用 24h）；或者改成 per-channel 配置的字段。

- **J5-MINOR-8**：voucher 签错（cumulativeAmount 写错）能不能"作废重签"？答案是**不能**——cumulativeAmount 单调递增，错签的 voucher 如果 amount 大于真实，payer 之后只能签更大值才能 settle 真实差额；但如果 voucher 被 payee 上链，payer 已经多付了。
  - 修复建议：SDK 强制使用 `signVoucher(channelId, currentCumulative + delta)` 模式，传入 delta 而非绝对值。前端永远显示 delta (本次结算金额)，而不是 cumulativeAmount。

- **J5-MINOR-9**：双方下线如何兜底？payer 离线 → payee 不能 close（需 payer 签字 finalAmount，或 payer requestClose）；payee 离线 → payer requestClose + 等 15 min withdraw。**payer 离线超过 N 天**：payee 需要"watchtower"来发起 close —— 协议无内建 watchtower。
  - 修复建议：文档化 watchtower 模式（外部服务），或在 MPC 增加"timeout-based force settle"路径（payer 长时间未活动 → payee 可强制结算 latest voucher）。

- **J5-MINOR-10**：channelId 由 `keccak256(payer, payee, token, salt)` 生成（推测；需 verify openChannel 实现）— 同 (payer, payee, token) 对希望同时开多通道，必须用不同 salt。SDK 若硬编码 salt=0，第二次开通道会 `ChannelAlreadyExists`。
  - 修复建议：SDK 默认 salt 用 `keccak256(timestamp_ms, random)`；前端对 ChannelAlreadyExists 提示 "您已有该 payee 的开放通道，是否补充？"。

**恢复缺失**：
- 通道资金锁住期间 payer 的 token 流动性损失。建议 SDK 在 UI 上明确显示"已锁仓 X，预期服务期 Y 天"。

### Section B — Codex 独立发现（本地审计漏掉的）

#### B-N1：未来时间戳 cache 可绕过 staleness 或 brick paymaster
- **位置**：`SuperPaymaster.sol:347`、`PaymasterBase.sol:474`、`PaymasterBase.sol:299`
- **严重度**：**HIGH**
- **现象 / 利用**：`SP.updatePriceDVT` 接受 `updatedAt` 在**未来**（仅拒绝"过旧"），导致 `validUntil = cachedPrice.updatedAt + priceStalenessThreshold` 可被任意推远 → staleness 检查事实失效，操控价格可"粘附"。另：`PaymasterBase.setCachedPrice` 同样允许未来 `timestamp`，进而 `postOp` 中 `block.timestamp - cachedPrice.updatedAt` **下溢 revert**，可能 brick 全部 op。
- **修复**：所有可设 cached timestamp 的路径强制 `updatedAt <= block.timestamp`（或 `<= block.timestamp + smallSkew`）；admin-set 路径加 monotonicity + timelock。

#### B-N2：Staking 退出费上限可被 `GTokenStaking.setRoleExitFee` 直接绕过
- **位置**：`Registry.sol:321`、`GTokenStaking.sol:373`
- **严重度**：**MED**
- **现象 / 利用**：`Registry.configureRole` 强制 `exitFeePercent <= 2000`（20%），但 `GTokenStaking.setRoleExitFee` 允许 `msg.sender == owner()` 设任意 `feePercent` 与 `minFee`，**完全绕过** Registry 的 20% cap。这破坏了治理约束，可在沉默中把 exit 变成接近全额没收。
- **修复**：去除 `GTokenStaking.setRoleExitFee` 的 owner 旁路，或 mirror Registry 的 cap 校验，或让 Registry 成为唯一可信调用方（带 timelock）+ 文档化。

#### B-N3：`updatePriceDVT` deviation 计算未防 `chainlinkPrice <= 0`
- **位置**：`SuperPaymaster.sol:361`
- **严重度**：**MED**
- **现象 / 利用**：deviation 分支若 `latestRoundData()` 成功返回但 `chainlinkPrice <= 0`（含 == 0），除法**revert** → DVT 价格更新被禁用，恰好在最需要 fallback 时失败。可避免的 oracle 边角 DoS。
- **修复**：deviation 分支前 `require(chainlinkPrice > 0)`，否则 skip deviation（或显式 OracleInvalid revert，但需保证有备用 rescue 路径）。

#### B-N4：blacklist BLS 消息缺 chain 绑定 / nonce → 跨链 replay
- **位置**：`Registry.sol:387`
- **严重度**：**MED**
- **现象 / 利用**：`updateOperatorBlacklist`（带 proof 时）签名消息为 `abi.encode(operator, users, statuses)`，**无 `chainid`、无 `proposalId`、无 nonce**。合法 proof 可跨链 replay（同 calldata），且无单次性语义。
- **修复**：消息加 `block.chainid` + `proposalId`/nonce，链上存按该标识 keying 的 replay-prevention bitmap；或路由到已绑定 `block.chainid` 的 DVT proposal 路径。

#### B-N5：`markProposalExecuted` 可预先 poison 任意 proposalId（DoS）
- **位置**：`DVTValidator.sol:59/73/86`
- **严重度**：**HIGH**
- **现象 / 利用**：`markProposalExecuted(id)` 对任意 `id` 设 `executed = true`（无存在性检查），`createProposal` 复用 `proposals[id]` storage 但**不重置** `executed=false`。`executeWithProof` 接受任意 `id`，攻击者只要能让 aggregator 调 `markProposalExecuted` 就可预 poison 未来 proposalId 让其永远不可执行（`executeWithProof` 立即 revert on `p.executed`）。结合 P0-5（无鉴权），任意 EOA 可达；即使修复 P0-5，恶意 validator 仍可 grief 治理。
- **修复**：`createProposal` 显式 `p.executed = false`（理想是清空全字段）；`executeWithProof` 要求 `id < nextProposalId`（或加 `exists` flag）；考虑只在 aggregator 跟踪 executed-ness，不在 DVTValidator 重复存。
