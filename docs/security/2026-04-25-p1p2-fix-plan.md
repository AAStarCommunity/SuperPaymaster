## 阶段二：PaymasterBase 修复（充裕空间，独立 PR）

### PR-2 `fix/p2-paymasterbase-cleanup`

**分支**: `fix/p2-paymasterbase-cleanup` → `main`
**来源**: B5-N2 + P1-20 + B5-N3 + B5-N4
**文件**: `contracts/src/paymasters/v4/PaymasterBase.sol`

| 来源 | 问题 | 修复 | size 影响 |
|---|---|---|---|
| B5-N2 | `getRealtimeTokenCost` 是死代码（Legacy 标注，0 外部调用） | 删除整个函数 | **约 -150 B** |
| P1-20 | `setPriceStalenessThreshold` 无范围限制（可设 0 或 100年） | 加 `require(v >= 60 && v <= 86400, "OutOfRange")` | +~30 B |
| B5-N3 | `postOp` 内 `catch {}` 静默吞没 oracle 更新失败 | 加 `emit PriceUpdateFailed(reason)` | +~20 B |
| B5-N4 | `setPriceStalenessThreshold` 无事件 | 加 `emit StalenessThresholdUpdated(old, new)` | +~15 B |

**净 size 影响**: 约 **-85 B**（负数，节省空间）
**风险**: 🟢 低。B5-N2 需要先在整个仓库 grep 确认 `getRealtimeTokenCost` 零调用。
**依赖**: 无。与所有其他 PR 独立。

## 阶段三：MicroPaymentChannel 修复（独立 PR）

### PR-4 `fix/p2-mpc-defensive`

**分支**: `fix/p2-mpc-defensive` → `main`
**来源**: B4-N8 + B4-N6 + B4-N5
**文件**: `contracts/src/paymasters/superpaymaster/v3/MicroPaymentChannel.sol`

| 来源 | 问题 | 修复 | size 影响 |
|---|---|---|---|
| B4-N8 | `openChannel` 不检查 `payee != payer`，可自我开通道锁定资金 | 加 `require(payee != msg.sender, "SelfChannel")` | +~15 B |
| B4-N6 | `requestCloseChannel` 可重复调用重置计时器（仅伤害自己，但逻辑混乱） | 加 `if (ch.closeRequestedAt != 0) return;` noop | +~10 B |
| B4-N5 | channel finalized 后 struct 永不清理，长期 storage 增长 | `closeChannel/withdrawChannel` 末尾加 `delete _channels[channelId]`（注：破坏历史查询，见 note） | +~10 B |

> ⚠️ **B4-N5 注意**：`delete _channels[channelId]` 会使 `getChannel(id)` 在 finalize 后返回 0 值而非历史记录。如果 SDK 依赖 `getChannel` 查询已结束通道，需先在 SDK 中迁移到事件索引。如不确定，可先只做 B4-N8 + B4-N6，B4-N5 延后。

**净 size 影响**: **+~35 B**（充裕）
**风险**: 🟢 低（B4-N5 降级为可选）
**依赖**: 无

## 阶段五：Registry 修复（需注意 241B 预算）

### PR-6 `fix/p1-registry-exit-cleanup`

**分支**: `fix/p1-registry-exit-cleanup` → `main`
**来源**: P1-32 (M-01) + P1-28 (B1-N1) + B1-N10
**文件**: `contracts/src/core/Registry.sol`

| 来源 | 问题 | 修复 | 估算 size |
|---|---|---|---|
| P1-32 | `exitRole` 仅清 COMMUNITY/ENDUSER 的 `roleMetadata` + `roleSBTTokenIds`，DVT/ANODE/KMS/PAYMASTER_* 退出后留脏数据 | 把 `else if (ENDUSER)` 分支改为 `else`（所有其他角色也做相同 delete） | **+~50 B** |
| P1-28 | `setStaking()` 不调 `_syncExitFees`，导致 exit fee 全 0（运维风险） | 调用前检查 `_syncExitFees` 是否存在；如无该函数则改为调用 `configureRole` loop | **+~40 B** |

**剩余 Registry headroom after this PR**: ~111 B

> ⚠️ **P1-28 依赖 `_syncExitFees` 是否存在**：如果 Registry 没有该函数，需要先在另一 PR 中实现它（或改用 loop 调用 `configureRole`）。执行前先 `grep _syncExitFees contracts/src/core/Registry.sol`。

**风险**: 🟡 中低。Registry 改动影响退出流程，需全量测试（`forge test --match-path contracts/test/v3/Registry.t.sol`）
**依赖**: 无

## 待确认事项

### Q1：agent policy 代码（`agentPolicies` / `setAgentPolicies` / `_resolveAgentPolicy`）处置方向？

当前状态：P0-15 仅注释掉了 `_applyAgentSponsorship` 的调用点（validation/postOp 未接入），但相关存储和函数仍保留在 SP 中。

**选项 A：删除**（推荐，为 SP 腾出 ~300B+ 空间）
- 从 SP 删除 `agentPolicies` mapping、`setAgentPolicies`、`_resolveAgentPolicy`、`getAgentSponsorshipRate` 等
- 同步从 `ISuperPaymaster.sol` 删除对应接口
- `__gap` 从 34 回到 38（4 个 storage slot 节省）
- 为 P1-4/P1-5/P1-39/P1-34-SP 等后续修复腾出空间

**选项 B：保留**（如果 V5.1 发布时间确定）
- SP 保持现状，P1-4/P1-5 等修复只能等其他地方腾出空间后再做
- PR-8/PR-9 无法执行

### Q2：B4-N5（`delete _channels[channelId]`）是否接受历史查询断点？

如果 SDK 当前依赖 `getChannel(id)` 查询历史通道，需要先迁移到事件索引再删除。否则直接加 `delete`。
